# 2026-06-04 - Diagnostic logging agents + Déploiement Railway Aminyo OS

## Problème : activity_logs vide (3 lignes de test seulement)

**Cause racine :** Aucun mécanisme central de logging d'activité. Chaque agent écrit dans son propre fichier `.log` local plutôt que dans Supabase.

### Schéma exact de activity_logs (vérifié)
- id (auto-increment)
- created_at (timestamptz, auto)
- agent (text)
- action (text)
- level (text) — info/warning/error
- details (jsonb, nullable)

Le schéma **accepte** des insertions — ce n'est pas un problème de RLS.

### Pourquoi les agents n'écrivent pas dedans

1. **Le seul agent qui a un logger Supabase intégré** est `comptable/logger-agent.js`, mais il n'est appelé que par le comptable lui-même
2. **Les autres agents** (auditeur, facteur, prospecteur, sentinelle, batisseur...) écrivent tous dans des fichiers `.log` locaux avec `fs.appendFileSync()`
3. **Le module `supabase-memory.js`** écrit dans `agent_memory` et `memory_episodic`, pas dans `activity_logs`
4. **Le `learning-orchestrator.js`** enregistre les actions mais pas dans `activity_logs`

## Solution créée : logger-agent.js comme mécanisme central

Fichier : `/data/.openclaw/plugin-skills/comptable/logger-agent.js`

### Comment ça marche
- Fonction `log(agent, action, level='info', details={})` qui insère dans `activity_logs`
- Fonction `getLogs(agent, limit=20)` pour lire les logs
- Fonction `getStats()` pour les stats dashboard
- Utilise `SUPABASE_SERVICE_ROLE_KEY` (donc bypass RLS)

### Prochaine étape
Intégrer cet appel `log()` dans chaque agent pour remplacer les `fs.appendFileSync()` locaux.

## Déploiement Railway — Aminyo OS API Bridge

**Service créé :** `aminyo-os-api` sur Railway
- URL : `https://aminyo-os-api-production.up.railway.app`
- Repo : `justmodjo/aminyo-os` branch `main`
- Variables d'env : `SUPABASE_URL` + `SUPABASE_SERVICE_ROLE_KEY` (liées au service)
- Build : `npm install && npm start` (Node.js Express)
- Routes : `GET /` (PWA), `GET /api/leads`, `GET /api/clients`, `GET /api/projects`, etc.
- Domaine Railway généré automatiquement

**Architecture sécurisée :**
- `service_role_key` uniquement côté serveur (Railway env vars)
- Frontend PWA appelle `/api/*` sur le même domaine
- Pas de clé exposée dans le frontend
